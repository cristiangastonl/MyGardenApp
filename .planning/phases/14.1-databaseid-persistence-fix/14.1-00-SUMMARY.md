# Phase 14.1 Plan 00: databaseId Persistence Fix Summary

**Surgical fix (~10 LOC across 2 files) that closes the user-visible verification gap surfaced during Phase 14 manual checkpoint. `AddPlantModal.handleAdd` and `PlantIdentifierModal.plantData` both now persist `plant.databaseId` when the saved plant maps to a catalog entry — `AddPlantModal` reads from `prefilledPlant.id` (when shape matches PlantDBEntry), `PlantIdentifierModal` resolves via `findPlantInDatabase(selectedPlant.scientificName)`. With this fix, `MyPlantDetailModal.strictDbEntry` resolves to the actual catalog entry for any PlantNet-identified plant whose species exists in the 64-entry catalog, and the 5 Phase 14 educational fields (careAction / placement* / whyRationale) plus override-detection (EDU-05) finally render real content end-to-end. Manual e2e verification on Android dev client deferred to next device-test session per v1.2 backlog (pre-existing pattern).**

## Decisions Made

1. **Treat as gap-closure (decimal phase) instead of Phase 14 polish.** The bug lives in upstream add-plant flows (authored in earlier phases ~v1.1 / Phase 7), not in the Phase 14 modal-restructure scope. Phase 14 made the consequence visible by introducing the 5 educational fields that depend on `databaseId` resolution. Marked `gap_closure: true` in plan frontmatter.

2. **Single atomic plan + commit.** ~10 LOC total, no new types/tests/UI work. Bypassing the full discuss/plan/research ceremony is appropriate: the bug + fix scope were fully documented in `v1_2_test_backlog.md` (authored during 14-08 root-cause analysis), and the verification surface (smoke runner + check:i18n-keys + tsc) is already exercised by Phase 14.

3. **Shape-check pattern for AddPlantModal.** Used the same `'id' in prefilledPlant && 'category' in prefilledPlant` predicate that AddPlantModal already uses at line 69 (for `inferWaterMode(cat, dbId)`). Safer than `instanceof` since prefilledPlant is a structural TypeScript type, not a class.

4. **`findPlantInDatabase` for PlantIdentifierModal.** Leverages existing fuzzy-match utility. Unmatched scientific names produce `undefined`, so:
   - Phase 12 unknown-plants tracker continues to silently log them (no regression)
   - The Plant is still saved with all other fields populated (typeId, lightLevel, etc.) — just without catalog-resolved educational content
   - User experience: empty-state placeholders shipped in Phase 14's commit `2ebdd43` provide graceful fallback ("Acá verás recomendaciones de cuidado y problemas activos para esta planta.")
