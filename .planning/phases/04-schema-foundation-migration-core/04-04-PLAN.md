<objective>
Refactor `notificationScheduler.ts` and the `isSensitiveToSun` heuristic in `plantInfo.ts` to consume `Plant.lightLevel` (the v1.1 field) instead of `Plant.sunHours` (the v1.0 field). This must land BEFORE the post-migration reschedule trigger (Plan 07) — otherwise the reschedule pass will throw on `undefined.sunHours` for any newly-created v1.1 plant and silently disable the entire notification subsystem via `markNotificationsUnavailable` (CRIT-3 from research).

Purpose: Sever the notification scheduler's dependency on the deprecated `sunHours` field. The scheduler currently reads `plant.sunHours` directly in `calculateSunWindow`, `groupPlantsBySunHours`, and `scheduleUVWarning` filter. After this plan, these all consume `lightLevel` with defensive fallbacks for the rare migration-failure-then-reschedule edge case.

Output: scheduler refactored; plantInfo helper updated; both files compile; tsc green; grep guard still green (both files retain allowlist status because the legacy reads are now defensive fallbacks, not primary code paths).

     * Calculates the optimal direct-sun window for a plant based on its lightLevel.
     * Returns null when the plant does not need a scheduled outdoor window

    function calculateSunWindow(
      plant: Plant,
      sunrise: Date,
      sunset: Date
    ): { start: Date; end: Date } | null {
      const hours = lightLevelToSunHours(plant.lightLevel);
      if (hours <= 0) return null;

      const start = new Date(sunrise.getTime() + SUNRISE_OFFSET_MINUTES * 60 * 1000);
      const sunHoursMs = hours * 60 * 60 * 1000;
      let end = new Date(start.getTime() + sunHoursMs);

      const maxEnd = new Date(sunset.getTime() - 30 * 60 * 1000);
      if (end > maxEnd) {
        end = maxEnd;
      }
      return { start, end };
    }
