<objective>
Wave 3 — migrate the 4 read-side consumers from cached-on-Plant-instance reads to live lookup via `getCatalogEntry(plant.databaseId)`. This is the runtime payoff for Phase 8: a typo fix to a `tip` in `plantDatabase.ts` propagates to every existing user plant on next render (Success Criterion 3 from ROADMAP Phase 8).

The migration pattern is uniform across consumers (RESEARCH §Consumer Migration Pattern, lines 488–520):

```typescript
// BEFORE: stale copy on Plant instance
<Text>{plant.tip}</Text>

// AFTER: live lookup with defensive ladder
const entry = plant.databaseId ? getCatalogEntry(plant.databaseId) : null;
const translated = entry ? getTranslatedPlant(entry) : null;
<Text>{translated?.tip ?? ''}</Text>
```

**Field name confirmation (Pitfall 4):** the field is `Plant.databaseId`, NOT `Plant.dbId`. CONTEXT.md uses "dbId" loosely; the correct field is verified at `src/types/index.ts` line 56. NO task in this plan writes `dbId` in code edits — always `databaseId`.

**`PlantDetailModal.tsx`** is the catalog-side modal (renders a `PlantDBEntry` directly, not a `Plant`). It should already be entry-driven — Task 3 only verifies and adds `getCatalogEntry` call if needed for canonical resolution.

**`PlantIdentifier/IdentificationResults.tsx`** is intentionally OUT OF SCOPE per RESEARCH §Read-Site Consumer Map: identification result is from PlantNet, not catalog — no migration there.

**`PlantDiagnosis/PlantDiagnosisModal.tsx`** is intentionally OUT OF SCOPE per CONTEXT: Phase 8 does NOT change diagnosis modal content (Phase 7 territory).

Purpose: deliver patch-update propagation (CAT-04) — the user-visible win of Phase 8.

Output: 4 component files migrated to live lookup; tsc green; smoke green.
