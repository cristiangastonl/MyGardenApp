<objective>
Phase 14.1 gap-closure: persist `plant.databaseId` when adding a plant through `AddPlantModal` (catalog-prefilled flow) or `PlantIdentifierModal` (PlantNet identification). Without this fix, every plant added through current UI flows lacked a `databaseId`, so `MyPlantDetailModal.strictDbEntry` always resolved to null and the 5 Phase 14 educational fields (careAction / placement* / whyRationale) could never render real content — they always fell back to the empty-state placeholders shipped in Phase 14.

Surgical fix: ~10 LOC across 2 files. Closes EDU-04 + EDU-05 user-visible verification (was DEFERRED in 14-08-SUMMARY pending this fix).
</objective>

<context>
Surfaced during Phase 14 manual device verification (Plan 14-08). After fixing the visible UX issues inline (animation, placeholders, copy renames, first-render glitch), the deeper architectural gap was identified: regardless of fast refresh + fixes, plants added through the existing UI never get `plant.databaseId` populated. Documented in `v1_2_test_backlog.md` Phase 14 entry. User opted to close Phase 14 and execute this gap-closure as Phase 14.1.

**Why this is gap_closure (not a Phase 14 in-scope task):** The bug lives in upstream add-plant flows (AddPlantModal + PlantIdentifierModal) authored in earlier phases (~v1.1 / Phase 7). Phase 14 made the consequence visible by introducing the 5 educational fields that depend on `databaseId` resolution. Phase 14's modal-restructure scope correctly didn't touch the add flows.

**Why a single plan (not a multi-wave phase):** ~10 LOC total, no new types, no new tests beyond smoke runner that already exercises the educational rendering path, no UI/UX work, no i18n. Treating this as a single atomic fix is appropriate given the small surface and clear root-cause documentation.
</context>
