<objective>
Two sequential manual-action checkpoints that the executor cannot perform but the user can:

1. **Task 1 (Deploy + initial secret set + first verification):** User runs `supabase functions deploy get-plant-care` and `supabase secrets set PERENUAL_API_KEY=<CURRENT key value>` (the same key that's currently in shipped APKs — we'll rotate in Task 3). Then verifies live: identify a known plant (e.g., monstera) in Expo Go, confirm Supabase Functions logs show the invocation succeeded.

2. **Task 3 (Key rotation + final secret update + final verification):** User logs into Perenual dashboard, generates a new API key, invalidates the old one, then runs `supabase secrets set PERENUAL_API_KEY=<NEW key>`. Then verifies live AGAIN: re-identify a plant, confirm the new key works.

(Task 2 is the "decision-gate" — confirms Task 1 succeeded before allowing Task 3 to proceed. Without it, the rotation could happen before deploy is verified, breaking production.)

Purpose: The CONTEXT.md ordering lock — "key rotation happens AFTER edge function is deployed and verified working with the new key in Supabase secrets. Otherwise we break production for the brief window between key invalidation and deploy." Plan 10-03 enforces this ordering with explicit checkpoints.

Output: ZERO files modified. Two `Deployed Function` confirmations (Tasks 1 and 3) and one Perenual dashboard rotation. The gap window between Plan 10-02 ship and Plan 10-03 Task 1 completion is the only period the live app loses Perenual enrichment — once Task 1 deploys + secret is set, enrichment resumes. After Task 3 rotation, only the edge function (with the rotated key) can reach Perenual; old shipped APKs (with the OLD bundled key) lose enrichment but degrade safely.

Why this is a multi-checkpoint plan instead of one big checkpoint: separating the deploy-verify and the rotation gives the user a clear abort point. If Task 1 verification fails (function logs show error), the user can fix the deploy WITHOUT having rotated the key, leaving shipped APKs functional. Once Task 3 rotates the key, there is no rollback for old APKs.
</objective>
