## Area: Plant Identification → LightLevelPicker

### 🟡 #1 — Picker intent ambiguous (reported 2026-05-02)

**What the user saw:** PlantNet identifies a plant correctly, then a screen labeled "Nivel de luz" shows 4 options to choose from.

**What's unclear:** Whether the user is:
- (a) **Selecting where they HAVE / WILL PUT the plant** (current behavior — the picker captures user's choice for THIS plant in THEIR space)
- (b) **Seeing a recommendation** of where the plant SHOULD go (passive, informational)

**User's preference:** Show a recommendation if possible (or both — recommended + user-overridable). At minimum, make it explicit that "estoy eligiendo dónde voy a poner mi planta" not "la app me sugiere dónde ponerla".

**Affected files:**
- `src/components/LightLevelPicker.tsx` — picker copy
- `src/components/PlantIdentifier/IdentificationResults.tsx` — section header / context above picker
- `src/i18n/locales/{en,es}/common.json` — `identification.lightLevelLabel`, `lightLevelHint.*`

**Possible v1.2 fix directions:**
- Add a clear question label above the picker: "¿Dónde tenés (o vas a tener) esta planta?" instead of just "Nivel de luz"
- Add a recommended badge on one of the 4 cards based on PlantNet's "sunlight" tag (currently only mapped server-side as a default seed — not surfaced as "recommended for this species")
- Or split into 2-step: "Esta especie prefiere [X]" → "¿Dónde la vas a poner? [picker]"

## Area: Identification entry points

### 🟡 #2 — Two plant identification entry points (reported 2026-05-02)

**What the user saw:** You can identify a plant in two places — once during Onboarding (when picking initial plants), and once inside the app (Plantas → Add → Identify with camera). Both flows do the same thing.

**Why it might be a problem:** Redundant / confusing — feels like two ways to do the same thing. May be intentional (onboarding's identification is the first plant), but worth reviewing whether to:
- Keep both (current — onboarding has identify-as-quick-start)
- Make onboarding identification clearly distinct (different copy, e.g., "Reconocé tu primera planta" vs "Identificar planta")
- Remove identification from onboarding (only catalog browse + manual add at first run, identification only available after onboarding completes)

**Affected files:**
- `src/screens/OnboardingScreen.tsx` — identification flow inside onboarding
- `src/components/PlantIdentifier/PlantIdentifierModal.tsx` — main identification modal

**Decision pending:** UX review — keep both with clearer differentiation, or consolidate.

## Area: Task generation for outdoor plants

### 🟡 #3 — "Sacar afuera" task on already-outdoor plants (reported 2026-05-02)

**What the user saw:** Identified a Geranio (which is an outdoor plant — category `exterior`). The plant detail correctly shows "Exterior" as the location/category. But in the task list, it shows "Sacar afuera" as a pending task — which makes no sense if the plant is already outside.

**Root cause (likely):** The `outdoor` task type in `getTasksForDay` is generated for any plant with a non-zero `outdoor` schedule, regardless of category. It's designed to remind users to take INDOOR plants outside periodically for sunlight (e.g., for cacti or plants that need direct sun). For plants whose `typeId` is already in `OUTDOOR_TYPE_IDS = {exterior, aromaticas, huerta, frutales}`, this task is meaningless and should be skipped.

**Affected files:**
- `src/utils/plantLogic.ts` — `getTasksForDay` outdoor-task emission branch
- Need to gate emission with `!OUTDOOR_TYPE_IDS.has(plant.typeId)` check (import from `lightLabel.ts`)
- Possibly also `src/utils/notificationScheduler.ts` if it schedules an outdoor reminder for these
- And the catalog entries themselves — Geranio + the 14 new outdoor entries should probably have `outdoor: 0` (no take-outside task) by default

**Severity consideration:** 🟡 polish (doesn't break the app, just confusing UX). Could be elevated to 🔴 blocker if user testing shows this generates daily noise tasks for the new 14 outdoor catalog entries — would mean every outdoor plant the user adds gets a useless task.

**Possible fix directions:**
1. **Code gate (defensive):** Skip outdoor-task emission in `plantLogic.ts` when `typeId ∈ OUTDOOR_TYPE_IDS`. Independent of catalog values.
2. **Catalog data fix:** Set `outdoor: 0` for all entries in OUTDOOR categories (Phase 8 may have left these at default values).
3. **Both** — defensive gate + data fix. Probably the safest combo.

### 🟡 #3a — Possibly related: picker label set for outdoor plants (reported 2026-05-02)

**What the user saw:** During identification of the Geranio (outdoor plant), the user picked "luz indirecta" — which sounds like the INDOOR label ("Luz brillante indirecta") rather than the OUTDOOR label ("Sol parcial"). Need to verify whether `LightLevelPicker` correctly switched to outdoor labels for the Geranio identification result.

**To verify:**
- Open identification flow with a known outdoor plant (geranio, jacarandá, romero)
- Check whether the 4-card picker shows "Sol pleno / Sol parcial / Semi sombra / Sombra" (outdoor labels) OR "Luz directa / Luz brillante indirecta / Luz media indirecta / Poca luz" (indoor labels)
- Per Phase 9 RESEARCH: `IdentificationResults` derives `typeId` from `plant.indoor === false ? 'exterior' : 'interior'`. If PlantNet returned `indoor === false` for Geranio, picker should show outdoor labels.

**Affected files:**
- `src/components/PlantIdentifier/IdentificationResults.tsx` — typeId derivation
- `src/utils/plantIdentification.ts` — `convertPlantNetResult` `indoor` field mapping from PlantNet response

**Possible fix:** If PlantNet didn't supply `indoor` info, the derivation defaults to `'interior'` (indoor labels). For known outdoor categories at the catalog level, prefer the catalog entry's category over PlantNet's indoor flag.

## Area: Plant detail — recommendation + education vs passive tracker

### 🔴 #6 — Detail modal doesn't explain WHAT to do, WHY, or RECOMMEND placement (reported 2026-05-02)

**What the user saw:** Tap a cactus card → detail modal. Sees `🤚 Chequeá cada 28d` and `Sol: Luz brillante indirecta`. Nothing else explains:
- HOW to check the soil ("hundí el dedo 5cm, fijate si está seca")
- WHY 28 days vs every week (cactus stores water, root rot risk if overwatered)
- WHERE to place the plant (which window, how close, direct vs filtered light)
- What "Sol pleno" actually means in practical terms (6+ horas sol directo, vs "Semi sombra" = 3-4 horas filtradas)

**The fundamental product reframe:**

> "No tenemos que perder de vista que nosotros tenemos que ser quienes recomendamos las cosas como deberían ser y luego el usuario mediante el chat vaya ajustando como las va a hacer."

The app currently treats the user as the source of truth (user sets lightLevel, waterMode, etc; app passively tracks). The user is calling for the inverse: **the app recommends from horticultural truth (catalog), the user adjusts to their reality via chat / settings**.

**This connects to UAT #1 (picker intent ambiguous):** the picker should show a *recommendation* derived from the catalog entry's natural lightLevel for that species, not start blank waiting for user input. User then either accepts the recommendation or overrides.

**Severity:** 🔴 — this is the difference between a tracker (Greg/Florish at their basic) and a guided plant assistant (Planta at its best, PictureThis premium). It's the highest-impact change in v1.2 because it shifts the value prop from "track what I do" to "tell me what to do, then track".

**What v1.2 detail modal should include (educational-first redesign):**

1. **"¿Qué hacer?" (action card)** — voseo, plain language:
   - Cactus: "Cada 28 días, hundí el dedo a 5cm de profundidad. Si la tierra está seca → regá hasta que el agua salga por los agujeros. Si está húmeda → esperá una semana más."
   - Houseplant fija: "Regá cada 5 días con ~200ml de agua. Dejá secar la primera capa entre riegos."

2. **"¿Dónde ponerla?" (placement card)** — recommendation + alternatives:
   - "Recomendado: junto a ventana sur (luz directa 6+ horas)."
   - "También funciona: ventana este (sol mañana suave) — vas a ver crecimiento más lento pero saludable."
   - "Evitá: pasillos sin luz natural — tu Monstera no va a sobrevivir más de 2 meses."

3. **"¿Por qué?" (rationale, opcional collapsable)** — short education:
   - Cactus: "Los cactus almacenan agua en sus tejidos. Riego excesivo = raíz podrida (la causa #1 de muerte de cactus en hogares)."
   - Monstera: "Originaria de la selva tropical, recibe luz filtrada por el dosel. Sol directo intenso le quema las hojas (manchas marrones)."

4. **"Tus ajustes" (current user state)** — confirmar lo que el usuario eligió + un "ajustar" tap:
   - "Vos elegiste: Luz brillante indirecta. Cada 5 días."
   - Si difiere de la recomendación: "Esto es diferente a lo recomendado para esta especie. ¿Querés ajustar?"

**Affected files (v1.2 scope):**
- `src/components/MyPlantDetailModal.tsx` — major restructure (current is info-pill style, needs sections)
- `src/data/plantDatabase.ts` — extend each catalog entry with new fields:
  - `careAction: { fixed: { ... } | soilCheck: { ... } }` — how-to instructions
  - `placementRecommended: string` + `placementAlternatives: string[]` + `placementAvoid: string`
  - `whyRationale: string` — short explanation of the recommendation
- `src/i18n/locales/{en,es}/plants.json` — translated content for all 64 entries
- New catalog content authoring (manual horticultural review) — significant lift but defensible diff vs competitors

**Effort estimate:** Medium-large. Code changes are 1-2 days. Content authoring (64 entries × 4 fields × 2 locales = ~512 strings) is the bulk — could be done with AI assist + horticultural review.

**Why this should be the v1.2 centerpiece:**
- Differentiates from "tracker" apps (Greg, Florish at base) by being a *guide*
- Connects to LATAM differentiation: Argentine horticultural advice in voseo with regional examples ("ventana al sur" means south = colder in southern hemisphere = different from US apps)
- Sets up future chat continuity: user reads the recommendation, asks AI "but my apartment is north-facing, what do I do?" → AI answers grounded in the catalog rationale
- Feeds into UAT #1 (picker recommendation) and #3a (outdoor labels) naturally
